# MVP 1 프로젝트 문서 (1-page)

## 드랍인 신청/안내 + 운영자 결제확정(승인) 운영

> WHAT/SCOPE는 `docs/spec-baseline-v0.3.2.md`가 단일 기준입니다.  
> 이 문서는 WHY/HOW(왜/어떻게 검증/운영할지)만 다룹니다.

---

## 1) Goals / Non-goals

### Goals

- 링크 기반 신청으로 드랍인/무료체험 흐름 표준화
- 운영자 승인(결제확정) 기반으로 오버부킹 0건 운영
- 신청 즉시 알림/목록 최신화로 “당일 현장 혼선(예고 없는 방문)” 감소

### Non-goals

- PG 결제/자동 환불/노쇼 자동화
- 통합 인박스(카톡/인스타/카페 메시지 통합)
- 신청자 자동 알림(SMS/Email) 기본 제공
- 신청자 자가 취소
- 현장 코치/리셉션 읽기 전용 권한(백로그)
- 멀티 지점/복잡 권한(백로그)

---

## 2) Success Signals (성공 신호)

### 정성(필수)

- 운영자: 드랍인/체험 관련 DM 응대 시간이 줄었다(주관 기록)
- 현장: 당일 “예고 없는 방문” 케이스가 줄었다(주관 기록)

### 정량(가능하면)

- 신청 → CONFIRMED 전환율
- 오버부킹 0건
- 신청 처리 리드타임(신청 생성 → 운영자 승인)

---

## 3) Pilot Plan (운영/배포 계획)

### 파일럿 대상

- 박스 1곳(운영자가 문제를 크게 느끼는 곳)

### 운영 프로세스(파일럿 기간)

- 운영자는 수시로 어드민에서 `PENDING_PAYMENT` 확인
- 입금 확인 후 승인 처리
  - 정원 여유 → `CONFIRMED`
  - 정원 초과 → `WAITLISTED` + 운영자 수동 안내(템플릿 복붙)
- 신청자는 조회 링크(토큰)로 상태 확인
  - 링크 불변(상태 변경 시 토큰 재발급 없음), TTL 30일

### 기간

- 1~2주(최소 운영 사이클 1회 이상 관찰)

### 사용자에게 보이는 필수 문구(완료 화면)

- “확정은 입금 확인 후 처리됩니다.”
- “조회 링크로 상태를 확인할 수 있습니다.”

---

## 4) Risks / Open Issues

### Risks

- 동시 승인으로 오버부킹 발생 가능 → 승인 처리 트랜잭션/락으로 방지 필요
- 스팸/악성 신청 → 최소 레이트리밋 필요
- 결제 확인 지연으로 운영 커뮤니케이션 비용 증가
- 외부 예약 채널 병행 시 정원 관리 혼선 가능성

### Open Issues

- 토큰 만료 이후 재발급 운영 방식(운영자 콘솔 “링크 재발급” 버튼 제공 여부)
- WAITLISTED 대기자 승격 자동화 범위(초기엔 운영자 수동으로 충분한지)
- 스팸 방지 수준(캡차 도입 여부 등)
